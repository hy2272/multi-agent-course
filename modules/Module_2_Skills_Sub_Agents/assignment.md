## Track 1 — Product Manager Assignment

### Objective

Imagine this is a product your team has built. The product concept, workflow, slide deck, and supporting materials have already been created by your team. Now, as the Product Manager, your task is to **create a website for the product** that can be used for MVP marketing, stakeholder communication, early user acquisition, or product validation.

The website should clearly communicate what the product does, who it is for, what problem it solves, and why users or customers should care.

### What You Need to Build

Create a website for the product — a landing page, MVP marketing website, product demo site, or early go-to-market website. It should include:

- **Product name and clear hero section** — a strong headline, short value proposition, and clear call-to-action.
- **Problem statement** — explain the problem this product is solving.
- **Target users or customers** — describe who the product is built for.
- **Product overview** — explain what the product does in simple, clear language.
- **Key features** — highlight the most important features or capabilities.
- **Workflow / how it works** — use the provided workflow document and slide deck to explain how the product works.
- **Benefits and value proposition** — explain why this product is useful and what value it creates.
- **Call-to-action** — e.g. "Request a Demo," "Join the Waitlist," "Try the MVP."
- **Design and user experience** — clean, understandable, and suitable for presenting to early users, stakeholders, or investors.

### Optional Enhancements

- Pricing section
- FAQ section
- Testimonials or placeholder customer quotes
- Demo video placeholder
- Screenshots or mockups
- Contact form
- Competitive positioning
- Product roadmap
- Blog or documentation section

### Submission

Submit your completed website as a `.zip` folder containing:

- All website files
- Any assets used (images, icons, logos)
- A short README explaining how to open or run the website
- Any assumptions you made while creating the website

## Track 2 — Developer Assignment

### Objective

As a developer, your task is to review all provided product materials, understand the product vision and workflow, examine the GitHub/codebase provided, and **implement at least one meaningful component** from Sprint Zero.

Sprint Zero refers to the early technical foundation of the product. Your goal is to build one core piece that helps move the product from concept toward a working implementation.

### What You Need to Build

Implement at least one core component of the product. This could include, but is not limited to:

- **Frontend component** — a key UI screen, form, dashboard, workflow page, or interactive component.
- **Backend component** — an API endpoint, database logic, authentication flow, processing function, or service layer.
- **Workflow implementation** — convert part of the workflow document into working functionality.
- **Data processing or AI component** — a model integration, prompt pipeline, data parser, recommendation logic, search function, or automation step.
- **Codebase improvement** — improve, refactor, optimize, or extend part of the existing codebase.
- **Prototype feature** — a functional prototype of one important product feature.

You are required to build **at least one** component. You are welcome to build more than one or attempt a larger implementation if you want.

### Expectations

Your implementation should show that you have:

- Reviewed the product description, workflow document, slide deck, and codebase
- Understood the purpose of the product
- Identified at least one technically meaningful piece to implement
- Written clean, understandable code
- Added documentation or comments where useful
- Made reasonable assumptions when details were missing

### Improvement and Optimization Suggestions

As part of your submission, include a short section explaining any improvement opportunities you noticed. You may mention:

- Product improvement ideas
- Technical architecture improvements
- UX improvements
- Workflow simplifications
- Code optimization opportunities
- Scalability concerns
- Missing features
- Risks or edge cases
- Suggestions for future development

This section does not need to be long, but it should show thoughtful review of the product and codebase.

### Submission

Submit your completed work as a `.zip` folder containing:

- Your implemented code
- Any modified or added files
- A README explaining:
  - What you built
  - Why you chose this component
  - How to run or test it
  - Any assumptions you made
  - Any improvements or optimizations you recommend
- Screenshots or demo video link, if applicable
